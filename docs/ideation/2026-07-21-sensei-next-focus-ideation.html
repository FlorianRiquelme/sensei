Codebase Context

sensei mines your own Claude Code transcripts for friction — corrections, tool-use denials, interrupts — via a deterministic, zero-token, stdlib-only Python mine.py. A nightly headless LLM stage clusters events and drafts config diffs; a morning /sensei review applies the ones you accept. The loop has grown fast — 16 issues closed, ADRs 0001–0015, and a prior ideation whose ideas already shipped (its Proof-of-Patrol Digest and in-session discovery became #16).

Levers unlocked

LOOP
Miner → cluster → propose → review/apply, with a headless allowlist enforcing "nightly never edits" (ADR 0001–0005).
MEMORY
Rejection = a cooldown, not a ban; stable proposal key for dedup (#1).
HABITS
repeat event type — mines re-supplied directives with no friction; wide stateless window (#9, ADR-0010/0011).
LADDER
A rule that keeps failing escalates to a proposed hook (#10, ADR-0012).
TEACH
Structured rejection verdicts + mini-grill: rejections now carry a reason (#12, ADR-0013).
PROOF
Deterministic nightly digest + in-session nudge — a missing digest is the failure signal (#16, ADR-0014/0015).
SHIP
Portable copy-install, testable miner, agent docs, triage labels (#2, #6, ADR-0006/0009).

Open edges (fertile ground)

GAP
Effectiveness is only seeded, never proven. #16 stores each accepted rule's pre-acceptance event count (baseline) — but nothing reads it back. No way to say "5 events → 0. Working," or to retire a rule that didn't.
GAP
Only learns from pain. Friction + repeats are negative/directive signals. Nothing learns from a smooth session or a skill that fired perfectly.
GAP
Proposal trust is thin. Config edits ride on LLM diligence — no dry-run, no measured blast radius, no rollback story past git.
GAP
Single-user, single-machine. "Built for distribution" (ADR-0006) has no adoption mechanic — a colleague starts cold; one person's hard-won rules can't seed another's.
GAP
The ladder tops out at "proposed hook." ADR-0012 itself asks: is there a rung above? And sub-threshold patterns have no stable identity across nights.

Topic Axes

  • AXIS 1

  • AXIS 2

  • AXIS 3

1

The Effectiveness Ledger — consume the baseline, prove each rule worked

Proof of impactConf 85%Cx Medium
Description
#16 already stores each accepted rule's pre-acceptance event count (baseline), and nothing reads it. Build the deterministic consumer: on every nightly run the miner recomputes each accepted rule's friction rate in the window after its acceptance date in decisions.jsonl, and writes a durable track record. /sensei status then renders one line per rule — "ddev-prefix: 5/wk → 0 since 2026-06-12. Working." vs "still 4/wk. Not working." Pure arithmetic over two artifacts sensei already produces — zero tokens, no new signal source. Each acceptance date is an intervention point; the before/after rate is a legitimate quasi-causal read.
Basis
directGrounding + the #16 plan: R10 stores the baseline but "no reporting, no delta math" — it is explicitly a seed with no consumer. decisions.jsonl carries date + key + verdict; the miner is stateless and recomputes over a wide window (ADR-0010). The join is all that's missing.
Rationale
This is the keystone. All five ideation frames independently landed here. Every downstream trust, autonomy, and sharing move is blocked on being able to say "this rule worked" — retirement (#5), quarantine (#7), the provenance in rule packs (#2), and confidence-weighted proposals all depend on it. It converts sensei from "propose and hope" into "propose and grade."
Downsides
Depends on the #16 baseline seed shipping first. Matching a rule to "its" future events is fuzzy under the current LLM-semantic clustering — accuracy improves markedly with the structured-trigger enabler (see the callout below the cards). A rule that's quiet because it works must not be misread as ineffective — pair with the clean-session signal (#4).
2

Portable Rule Packs — export proven rules; ship a starter pack

Distribution & adoptionConf 82%Cx Medium
Description
A stdlib export/import: bundle accepted rules (rule text + evidence + track record from #1) into one portable file. Import re-enters each rule into the recipient's /sensei review queue as an ordinary proposal — never auto-applied — so nightly-proposes/review-applies (ADR-0002/0005) stays intact. Two payoffs from one format: (a) team sharing — one engineer's hard-won CLAUDE.md rules seed a colleague's; (b) day-one value — ship a curated maintainer starter pack so a brand-new user's first review isn't an empty screen.
Basis
directGrounding names both gaps this bridges: "single-user, single-machine — no way for one person's hard-won rules to seed a colleague's," and "distribution is ready but has no adoption mechanics." ADR-0006 declares sensei "built for distribution"; ADR-0009's copy-not-sync model extends naturally from the tool to the tool's output. Import-as-proposal is the only team model that creates no new apply surface.
Rationale
All five distribution-touching frames converged here. sensei's real product isn't the miner — it's the accumulated judgment a long-running instance earns. Today that value is trapped on one machine. A pack turns it into something a team compounds on, with no server and no invariant broken.
Downsides
Evidence quotes can carry sensitive transcript text — redaction must be deliberate. A maintainer starter pack introduces opinionated defaults someone has to curate. The "track record" provenance is only meaningful once #1 exists (packs work without it, but weaker).
3

Counterfactual Dry-Run — show the moments a rule would fire, before you accept

Proposal quality & trustConf 80%Cx Medium
Description
At review time, before the accept/reject question, deterministically replay the proposed rule's trigger across the already-mined history and show the concrete past moments it would have changed — each with the verbatim quote and project the miner already captured. A narrow rule matches its seed events and little else; an over-broad rule lights up unrelated moments, so scope creep reveals itself at the decision point instead of degrading the config silently. A zero-token --dry-run for CLAUDE.md edits.
Basis
reasonedGrounding: proposal verification is "thin — no dry-run, no measured confidence." Proposals already carry 2–3 tagged verbatim quotes (SKILL.md step 4, verified), and the events are a deterministic set, so replay costs nothing. The pattern is borrowed from linters and VCS: --fix --dry-run, git rebase --dry-run — good tooling never applies blind, it previews the sites it will touch.
Rationale
Converts "sounds reasonable, accept" into an evidence-based click. Showing real historical blast radius is exactly what makes any future auto-apply (deliberately deferred, see rejections) ever thinkable — trust has to become mechanical before autonomy can grow.
Downsides
Matching a prose rule to events deterministically is the hard part — the same fuzzy-match issue as #1, strongest when paired with structured triggers. Adds a step to review; mitigate by showing the preview only where a trigger is inferable. A post-apply structural check (duplicate-heading / line-delta sanity) is a cheap companion that belongs in the same build.
4

Learn from Clean Sessions — a positive signal that vetoes over-broad rules and reinforces what works

Signal breadthConf 65%Cx Large
Description
Today the miner emits only friction. Add a deterministic positive signal: a session (or the span after a skill/tool fires) with substantial activity and zero friction is a "clean"/"win" event. Nightly uses it two ways — defensively, to veto or down-weight a proposal that would constrain behavior clean sessions routinely perform without complaint (kills autoimmune over-broad rules); and generatively, to draft positive proposals ("the tdd skill fires and sessions go smooth — sharpen its trigger so it fires earlier").
Basis
directGrounding gap, verbatim: "nothing learns from what went well — smooth sessions, a skill that fired perfectly." Confirmed in mine.py: only rtype=="user" becomes an event; assistant/tool activity is stored as context only (line 86). Structurally this mirrors the shipped repeat event (ADR-0011) — a positive twin of an existing mechanism. Analogy: immune self-tolerance (learn "self" so you don't attack healthy tissue) and clinical responder coding.
Rationale
sensei's entire worldview is pain; a nervous system that only feels pain eventually over-corrects — proposing fixes for things that already work. This is the cheapest defense against the scariest failure mode (silently making Claude worse) and it roughly doubles the learnable surface, shifting sensei from nag-detector toward coach.
Downsides
Highest build cost in the set — a new event type plus reliable skill/tool-invocation detection in the transcript. "Clean" is heuristic: a session can be quiet because the user gave up, so the signal needs guardrails before it's trusted to veto.
5

Rule Retirement via spaced reaffirmation — sensei's missing delete key

Autonomy & the human loopConf 68%Cx Medium
Description
sensei only ever adds prose; a self-improving system must also shrink. Once #1 exists, treat each accepted rule like a spaced-repetition card: if its post-acceptance friction stayed gone, the rule "graduates" and its next check-in pushes further out (less attention over time); if friction recurs or the baseline never moved, the interval collapses and nightly drafts a retire: proposal — walked through review like any other, reversible via git. Acceptance becomes provisional with a symmetric "prove-in" window, the mirror of the 30-day rejection cooldown.
Basis
reasonedGrounding: "no way to retire a rule that didn't work." CLAUDE.md loads into every session, so dead rules carry a real context cost against the project's own simplicity ethos. ADR-0012 already establishes that a rule's lifecycle is sensei's business (the escalation ladder) — retirement is the missing downward rung. The expanding-interval schedule is Anki's SM-2 inverted: a well-retained item is shown less, resurfacing only when it slips — which is exactly the "prove it works without nagging" mechanism.
Rationale
The first mechanism that lets sensei reverse its own past decisions — what "self-improving" actually requires — and it keeps CLAUDE.md lean instead of growing monotonically into noise. Attention flows only to the rules that are actually slipping.
Downsides
Strictly depends on #1. The central false-positive — retiring a rule that's quiet because it works — is precisely what the clean-session signal (#4) guards, so this is strongest built after #4.
6

Recall Leak Counter — make the miner audit its own blind spots

Signal breadthConf 88%Cx Low
Description
mine.py silently discards signal in four places and surfaces none of it: malformed JSONL is swallowed by a bare except: continue (line 68), events past MAX_PER_SESSION are dropped (line 95), the global MAX_TOTAL=400 truncates (line 164), and unreadable files are skipped (line 59). Add _meta: {parse_errors, capped_sessions, total_capped, unreadable_files} to events.json and surface it in the nightly digest — "sensei may be missing signal in project X." Small, stdlib, deterministic.
Basis
directVerified by reading the file: lines 66–69, 95–96, 164, and 59–60 each drop data with no trace, and the output dict (lines 166–171) carries no such counts. ADR-0004 stakes the whole design on the miner favoring recall — but recall is only a guarantee if you'd notice when it silently fails.
Rationale
The miner is the foundation every downstream stage trusts — clustering, and the effectiveness scoring in #1. A corrupted transcript or an unusually long session currently degrades signal invisibly. This is the cheapest possible guard on the project's core premise, and a clean quick win to ship first while the larger directions are scoped.
Downsides
Almost none — tiny surface, on-brand. Marginal alarm-fatigue risk if the counts are noisy; mitigate by only surfacing above a threshold.
7

Quarantine — the rung above the hook is out of the loop, not deeper in it

Autonomy & the human loopConf 62%Cx Medium
Description
When a pattern keeps qualifying but its rule and its escalated hook keep failing or getting rejected, stop re-proposing it. Move it to a durable "chronic friction" ledger surfaced once as "this needs a human design decision, not another rule" — a short structured writeup, not a diff. The ladder's top rung becomes escalation out of automation, and those recurring sub-threshold patterns finally get the stable cross-night identity they lack.
Basis
reasonedGrounding poses this exact open question: "the ladder tops out at 'proposed hook' — is there a rung above? Permissions were scoped out (ADR-0012)," plus "sub-threshold patterns have no stable identity across nights." The pattern is CI flaky-test quarantine: mature CI stops auto-retrying a test it can't settle, isolates it, and reports it for human root-cause. Quarantine respects the permissions scope-out — the terminal move hands the human a question, not autonomy.
Rationale
Prevents the single fastest uninstall trigger — sensei re-proposing what you keep rejecting — and names the class of problem that rules genuinely can't fix. It turns nagging into an honesty valve.
Downsides
Needs the escalation/hook machinery (ADR-0012) in place first, and "keeps failing" is only objective once the effectiveness read (#1) exists. A narrower variant — escalating to a proposed regression test — fits projects that already have a test suite.
Cross-cutting enabler

Structured rule triggers. Ideas #1, #3, and #5 all hinge on matching a prose rule to "its" events — today a fuzzy, LLM-semantic guess. Letting a proposal optionally carry a small machine-checkable trigger (target glob / tool name / keyword) alongside its prose would make that match exact and give sub-threshold patterns the stable identity #7 needs — the deterministic miner could then measure the effect of prose without the LLM ever entering the measurement loop. It's a substrate, not a standalone direction; worth deciding early because it changes how #1/#3/#5 are built.
