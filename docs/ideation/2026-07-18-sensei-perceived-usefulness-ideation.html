Grounding Context

Codebase context. sensei is a nightly macOS launchd agent: a deterministic, stdlib-only miner (mine.py) reads Claude Code transcripts and emits friction events (corrections, tool denials, interrupts); a skill clusters them headlessly at 05:30 (/sensei nightly, which writes proposals only and physically cannot edit config) and an interactive morning mode (/sensei review) applies what the user accepts. Rejections get a 30-day cooldown under a stable dedup key. Trust is architectural (ADRs 0001/0002/0005: deterministic miner, nightly-proposes-review-applies, allowlist enforcement); the miner favors recall and the LLM owns precision (ADR-0004); the project is explicitly built for distribution (ADR-0006).

The gap the grounding surfaced. Steady-state feedback loops are well designed (issues #9, #10, #12, #13 extend them), but everything the user actually sees is thin: day one is silent by construction (--days 1 window plus a ≥2-session threshold), a zero-proposal morning is indistinguishable from a silently failed run (the launchd && chain aborts without notification), accepted rules leave no visible track record, and discovery of new proposals rests entirely on an ephemeral 05:30 notification. Detection is engineered; perceived value is not.

External context. Near-identical prior art exists (agent-improvement-loop, claude-improve, agent-retro, LearnSkill); the convergent lessons: concrete evidence per proposal, promote existing rules before creating new ones, separate durable lessons from one-offs. sensei’s rejection cooldown is a differentiator none of them implement. Dependabot studies show advisory bots die of notification fatigue, not miss rate. The Copilot trust-calibration research names three failure modes to design against: abandon, verify-everything, blindly-accept. Habit research (Fogg): the morning ritual must cost ~1–2 minutes, and silence reads as a dead product. Evidence dossiers from five axis scouts live under /tmp/compound-engineering/ce-ideate/a7f3c2d1/.

1

The Founding Report: install-time backfill over full history

axis first-run-cold-startconfidence 85%complexity Low
Description
Make install.sh end with sensei’s first real output, not a promise: run mine.py --days 0 (or a ~30-day window) over the transcript history that already exists, kick off one headless nightly pass, and print “Founding report ready” with the proposals path. Fold in the transparency demo: show the events summary (“scanned 41 sessions, 87 events across 6 projects”) and point at events.json: “open it; this is everything sensei will ever show the LLM.” The realistic install base is existing Claude Code users, so the richest corpus sensei will ever see is sitting there at install time and is currently thrown away.
Basis
direct: install.sh:31 ends with only “sensei: installed. Runs daily 05:30 via launchd…”; mine.py:9 documents --days 0  # all time yet nothing in the shipped flow ever invokes it; the plist runs --days 1, so the first scheduled run scans a near-empty window. Verified by the Phase 3 spot-check. Five ideation frames independently converged on this, the strongest convergence in the run.
Rationale
Day-1 silence is not bad luck; it is a deterministic side effect of a steady-state window default that was never special-cased for install day. The first minutes after install are where “is this useful?” is decided, and the capability to answer it already ships as an unused flag. This converts “trust me, wait a day” into “look what it already found in your history.”
Downsides
Install now spends Claude tokens on an unrequested run and takes minutes instead of seconds; window size needs a decision (all-time may over-surface stale friction; 30 days is a safer default); running claude -p from an installer needs a clean failure path when the CLI isn’t authenticated.
2

Proof-of-Patrol Digest: the quiet morning proves the patrol happened

axis visible-payoffconfidence 80%complexity Medium
Description
Reframe the daily artifact: it is always a friction digest (sessions scanned, events by type and project, patterns brewing below threshold, accepted rules with no recurrence), and proposals are one section that is sometimes empty. The notification always carries the payload (“scanned 6 sessions, 14 events, 0 proposals; 3 rules holding”), never a bare “0 proposals.” The miner emits the deterministic baseline digest before the LLM stage runs, so even a silently aborted && chain leaves a dated artifact; a missing stamp is itself the failure signal (the night-watchman’s watchclock: an uneventful night must be distinguishable from a guard who slept).
Basis
direct: skill/SKILL.md:74-77: zero-pattern nights write a one-line file and notify “N proposals” with N often 0 (verified). external: habit research in the grounding (“‘0 proposals today’ silence reads as dead product; always surface something”) and morning-digest products: the ritual is the product.
Rationale
Most mornings of a well-tuned sensei are zero-proposal mornings (that is the success state), so the zero-morning surface is the most-seen surface of the product and currently the weakest. It also cannot distinguish health from silent failure today. One reframe fixes the highest-frequency touchpoint and the worst trust hole at once.
Downsides
Moving digest generation into the miner redraws the miner/LLM responsibility line from ADR-0001 (the miner would now write a human-facing artifact, not just events); a daily digest nobody reads can itself become notification noise if it isn’t glanceable.
3

Track-Record Spine: provenance stamps, friction receipts, /sensei status

axis visible-payoff · trust-auditabilityconfidence 75%complexity Medium
Description
One ledger-aggregation build powering three visible features. (a) Plant tags: review stamps every applied rule with provenance (<!-- sensei:2026-07-18 key=… -->), making “which config came from sensei, and which accepted decisions still survive in config” greppable in both directions. (b) Friction receipts: at accept time, store the pattern’s pre-acceptance event count in decisions.jsonl (one integer of new state); after the ~14-day grace, report the delta: “Rule ‘ddev-prefix-artisan’: 5 events → 0. Working.” (c) /sensei status: one screen showing last run and outcome, the allowlist actually in force, event/proposal/decision counts, acceptance ratio, and active cooldowns. The same primitive feeds issue #10’s escalation check.
Basis
direct: ADR-0012 (proposed) names the honest signal itself (“the honest signal would be ‘friction rate didn’t drop,’ but that needs a stored pre-acceptance rate to compare against — new state”) and declines it; SKILL.md:24’s decisions ledger already holds every verdict with a date, used today only for cooldown lookups; SKILL.md:92-93 applies rule text with no attribution. Verifier caveat honored: the dedup key exists only post-drafting, so receipts track accepted proposals, not arbitrary sub-threshold patterns.
Rationale
“Did sensei ever actually help?” is the retention question, and today the only answer is the user’s memory. Receipts are the one mechanism in the whole candidate set that lets sensei say “this rule worked” with a number; stamps make that computable and make every sensei-authored line individually attributable and revertable; status is where all of it becomes visible on demand.
Downsides
Stamps in user-owned files are a boundary the ADRs haven’t ruled on, and they can drift stale when a human hand-edits the rule; receipts add the first cross-run state beyond the ledger; a track record that shows mostly “no change” could undercut rather than build confidence.
4

Two-Minute Morning: consent-agenda review + in-session discovery

axis morning-reviewconfidence 75%complexity Medium
Description
Two changes to the ritual. Consent agenda: review opens with a one-line-per-proposal docket (title, target file, strongest quote); the user gives a batch verdict (“accept all except #3; let’s talk about #3”) and only pulled items get the full one-at-a-time walkthrough. Extension once trust accrues: proposal classes with N consecutive accepts get offered as one-keystroke batch consent (still human-gated; the ADR-0002 bright line holds). In-session discovery: a SessionStart hook prints “sensei: 3 proposals waiting (oldest 2 days), run /sensei review” the next time the user opens Claude Code, anchoring discovery to where review actually happens instead of a 05:30 notification that evaporates before breakfast.
Basis
direct: skill/SKILL.md:87-91: “Walk proposals one at a time… Ask: accept / reject / edit?” (verified); the plist’s 05:30 osascript notification is the only discovery surface. external: Fogg habit research (the ritual must cost ~1–2 minutes) and the approval-queue pattern’s demotion ladder (N consecutive approvals → lighter review).
Rationale
Serial walkthrough cost scales linearly with proposal count, so good weeks, when sensei finds the most, punish the user hardest, exactly when the habit should be reinforced. And an undiscovered proposal is indistinguishable from a dead tool. “Review over coffee” is the product’s own promise (README); this is what keeps that promise at volume.
Downsides
Batch consent trades scrutiny for speed; it needs an explicit answer to whether one keystroke over three items still satisfies “the human accepted each one.” A SessionStart hook adds an install surface and must never nag (once per day at most). There is tension with issue #12’s mini-grill, which adds turns to the same interaction this idea is shrinking.
5

Exercisable Trust: sensei why, loud failure, versioned contract

axis trust-auditabilityconfidence 70%complexity Medium
Description
Turn “trust as architecture” into something the user can exercise. sensei why <key> (or mine.py --trace): one command walks a proposal from its quoted evidence to the decision history, the underlying events (timestamp, project, session), and the exact transcript file each quote came from. That is a ten-second spot-check instead of grepping JSONL by hand, and the honest defense against the one real gap in the evidence chain: the LLM stage copies quotes into proposals and could paraphrase or hallucinate one. Plus: a || failure branch in the plist chain so a broken run fires a distinct “nightly FAILED” notification (broken must never look like quiet), and a schema_version field on events.json/decisions.jsonl before distribution creates independent readers of those files.
Basis
direct: every mined event already carries ts/project/session (verified in mine.py); the join to transcripts is mechanical, just unexposed. sh.sensei.plist.template:20-21’s && chain aborts silently on miner failure (verified). external: Copilot trust-calibration research: verification overhead is what erodes trust; cheap spot-checks are the fix for the verify-everything failure mode.
Rationale
Verbatim quotes are trust claims; a working provenance trace is trust verification. The first fabricated-looking quote a user can’t trace is the last proposal they’ll believe. And for a tool that reads private transcripts, being able to exercise the guarantees daily is what moves a new user from verify-everything to spot-check-occasionally.
Downsides
Mostly interface work but spans three small pieces; traceback needs quote-fragment matching against large JSONL trees to stay fast; schema versioning is invisible until a second reader exists (it earns its place here only bundled with the visible pieces).
6

Deterministic Quality Floor: backtest scores + pre-LLM duplicate suppression

axis proposal-qualityconfidence 60%complexity Medium
Description
Push two quality guards into the free, auditable, deterministic layer so perceived quality stops swinging with LLM variance. Backtest coverage: every proposal gets a deterministic score (“this rule would have addressed 7 events across 4 sessions in the last 30 days”), computed by re-scanning the wide-window events; proposals below a coverage floor are demoted to an appendix, and the number doubles as a triage signal in review. Duplicate suppression: a cheap deterministic pass greps the target file for near-matches of a candidate’s key/slug before the LLM drafts, killing the “proposed the same rule twice” failure outright instead of re-judging it non-deterministically every night.
Basis
direct: SKILL.md:68-70: evidence today is 2–3 verbatim quotes, proof of existence rather than magnitude; SKILL.md:59-61: non-duplication is an LLM judgment call made fresh every run (both verified). external: Dependabot studies: low-value churn, not miss rate, is what gets advisory bots deprecated.
Rationale
Users judge sensei by its worst proposal, not its best. This extends the project’s own founding split (ADR-0001/0004: miner owns reliability, LLM owns judgment) onto the quality axis; the checkable failure modes move to the layer that can’t hallucinate. The verifier rated this the most architecturally-grounded synthesis in the set.
Downsides
A coverage threshold is a new tunable that can wrongly bury a high-severity single event (the existing high-severity path must bypass it); slug-based near-matching in config files will have false positives/negatives at the margins; introduces quantitative machinery into a deliberately simple pipeline.
7

Speak My Language: lexicon packs + language-gap self-check

axis first-run-cold-start · distributionconfidence 65%complexity Low
Description
Extract the correction lexicon from a hardcoded regex in mine.py into drop-in data files (lexicons/en.txt, lexicons/de.txt, …) that the miner loads and unions at runtime, so one community contribution covers every future adopter in that language, with no Python edit and no private fork. Pair it with a self-check: when the miner sees a healthy session count but near-zero correction events, it says so plainly (“your language may not be covered; add a lexicon pack”) instead of failing exactly like a quiet week.
Basis
direct: README: “To detect push-back in your language, edit CORRECTION_RE in mine.py… and re-run install.sh”; ADR-0006 names distribution as the project’s governing constraint; corrections are the dominant friction signal (CONTEXT.md), so for non-EN/DE users the primary signal silently degrades with no way to tell “found nothing” from “can’t see my language.”
Rationale
This is the sharpest silent value-destroyer for the distributed audience the user’s question names (“people”): the tool fails invisibly for anyone outside two languages, and the documented fix is never triggered by anything the tool itself notices. Data files keep the stdlib-only constraint intact while turning a personal-config artifact into a community surface.
Downsides
Regex quality varies by contributor and language morphology (recall-over-precision helps; the LLM filters downstream); the self-check heuristic needs care to avoid nagging monolingual-English users with quiet weeks; small install-surface change (copy the lexicon dir).
